🧱 AOSP Architecture – The Android Operating System

The Boot Sequence: init, Zygote, system_server

After the kernel loads, the `init` process parses init.rc and starts native daemons (servicemanager, hwservicemanager, vold). Zygote pre‑loads framework classes and forks every app process. system_server hosts critical services like ActivityManagerService, PackageManagerService, and WindowManagerService.

# View running services
adb shell dumpsys activity services

HAL & Native Daemons

The Hardware Abstraction Layer (HAL) runs in its own process (or in‑process with the client). Attackers target privileged daemons (e.g., mediacodec, hwcomposer) for kernel escalation. SurfaceFlinger handles compositing; compromising it gives full screen capture.

# List HAL services
adb shell lshal

Framework Services & PackageManagerService

PackageManagerService verifies APK signatures, manages permissions, and handles install flows. Abusing its Binder interface can install apps silently or bypass signature checks.

⚡ ART Runtime – The Execution Engine

oat/vdex, JIT, and AOT Compilation

ART compiles DEX bytecode to native code (oat). VDEX stores uncompressed DEX for quick verification. JIT profiles optimize hot methods. Attackers can modify oat files to inject code at the native level.

# Dump ART method mappings
adb shell cmd package compile -m speed -f com.target.app

Method Hooking & Inline Caching Abuse

ART maintains a method table in the `ArtMethod` structure. Hooking the entry point (via Frida or custom native code) intercepts every invocation. Inline caching can be poisoned to redirect calls.

// Frida: find ArtMethod structure
var ArtMethod = ...;
Interceptor.replace(ArtMethod.entry_point, new NativeCallback(...));

Garbage Collection & Runtime Optimizations

Understanding GC pauses and how ART manages object allocation helps exploit timing windows or leak references. Use `adb shell dumpsys meminfo` to profile runtime memory.

🔗 Binder IPC – The Heart of Android

Binder Transaction Internals & Parcel Serialization

Every IPC call is a Binder transaction. The kernel driver (`/dev/binder`) copies data between processes. Parcel objects are serialized and reconstructed. Race conditions in Binder can lead to use‑after‑free or double‑free vulnerabilities.

# Watch Binder transactions in real‑time
adb shell cat /d/binder/transactions

Service Manager Abuse & Capability Leaks

The Service Manager (`servicemanager`) registers all system services. If an attacker can register a fake service with the same name, clients unknowingly connect to a malicious implementation. Binder tokens act as capabilities; stealing a token grants unauthorised access.

# List registered services
adb shell service list

Binder Fuzzing & Driver Exploitation

Fuzz Binder calls with malformed Parcel data. Kernel bugs in the Binder driver can lead to privilege escalation. Use tools like binder-fuzz or android-fuzzing.

🔒 SELinux & App Sandbox – The Policy Fortress

SELinux Domains & Policy Transitions

Every process and file has an SELinux context. Policy rules define which domains can access which resources. Attackers look for misconfigured transitions (e.g., `untrusted_app` to `system_server` via Binder).

adb shell getenforce
# View policy
adb pull /sys/fs/selinux/policy
sesearch -A policy

Policy Bypasses & Sandbox Escapes

App sandboxes isolate file access. Escaping the sandbox often requires exploiting a vulnerable system service or a kernel bug. Dirty Pipe (CVE‑2022‑0847) allowed writing to read‑only files, bypassing SELinux entirely.

# Dirty Pipe exploit on Android (conceptual)
./dirtypipe /data/local/tmp/rootfile /etc/passwd

💥 Android Kernel Exploitation

Use‑After‑Free & Race Conditions in the Kernel

Android kernels are based on Linux LTS. Vulnerabilities in the Binder driver, ION allocator, or GPU drivers are prime targets. A UAF in `binder_ioctl` can grant arbitrary kernel read/write.

Privilege Escalation Chains & KASLR Bypass

Combine a kernel bug with a KASLR leak (e.g., reading /proc/kallsyms) to build ROP chains. Common escalation paths: `ptrace` abuse, `suid` binaries, or kernel module injection.

Driver & Vendor Kernel Exploitation

Qualcomm, MediaTek, and Samsung kernels often include closed‑source drivers with known vulnerabilities. Fuzz the `/dev/kgsl-3d0` or `/dev/mtk_cmdq` interfaces.

🔐 ARM TrustZone & TEE Security

Keymaster, Gatekeeper, and StrongBox

Android Keystore uses Keymaster HAL, which communicates with the Trusted Execution Environment (TEE). StrongBox is a dedicated tamper‑resistant hardware module. Compromising the TEE (e.g., via a bug in QSEE) allows extraction of supposedly hardware‑protected keys.

# Check Keymaster version
adb shell getprop ro.hardware.keystore

QSEE & TEEGRIS Exploitation

Qualcomm's QSEE (Secure Execution Environment) has been exploited multiple times (e.g., CVE‑2020‑36737). Samsung's TEEGRIS is also a target. Exploits typically require a kernel compromise first, then escalate into the secure world.

🔒 Android Verified Boot & dm‑verity

vbmeta, Boot Chain, and Rollback Protection

Android Verified Boot (AVB) uses vbmeta to verify each partition (boot, system, vendor). Rollback protection prevents downgrading to a vulnerable version. Attackers exploit OEM implementation flaws, such as missing verification or weak keys.

# Check AVB status
adb shell avbctl get-verity

Unlocking Bootloader & Custom Verified Boot

Unlocking the bootloader disables verified boot, but the device is then marked as "orange state" (untrusted). Magisk patches the boot image to inject root while preserving AVB to a degree.

🛡️ Play Integrity & SafetyNet

Hardware vs Software Attestation

Play Integrity provides three evaluation types: BASIC (software), HARDWARE_BACKED (TEE), and STRONG (StrongBox). Apps like Google Pay require HARDWARE_BACKED. MagiskHide/Zygisk can bypass basic checks, but hardware attestation is extremely difficult to spoof.

# Test Play Integrity
adb shell am start -a android.intent.action.VIEW -d https://play.google.com/store/apps/details?id=com.scottyab.safetynet.sample

Magisk Zygisk & Universal SafetyNet Fix

Zygisk injects into Zygote to modify system‑level behavior. Modules like USNF (Universal SafetyNet Fix) spoof keystore responses and hide root from apps. This cat‑and‑mouse game is constant.

🧬 JNI & Native Library Reversing

JNI Internals & ARM64 ELF Analysis

Java Native Interface (JNI) allows calling C/C++ code from Java. The `libnative.so` file is an ELF binary. Use Ghidra or IDA to reverse it. Look for unsafe JNI functions like `NewStringUTF` that can be hooked.

# Extract native library
adb pull /data/app/com.app/lib/arm64/libnative.so
file libnative.so

Native Hooks & ptrace Anti‑Debugging

Frida can hook native functions via `Interceptor.attach`. Apps detect debuggers via `ptrace(PTRACE_TRACEME, ...)`. Bypass with Frida's `--realm=emulated` or by patching the native code.

// Frida hook native function
var func = Module.findExportByName("libnative.so", "check_root");
Interceptor.attach(func, { onLeave: function(retval) { retval.replace(0); } });

🦠 Android Malware – Banking Trojans, RATs, Spyware

Accessibility Service Abuse & Overlay Attacks

Banking trojans like Anubis and Cerberus use Accessibility Services to perform automatic clicks, steal credentials, and intercept OTPs. Overlay attacks show a fake login screen above the legitimate app.

// Accessibility service config
<accessibility-service android:canPerformGestures="true" android:canRetrieveWindowContent="true"/>

Persistence & C2 Communication

Malware persists via BroadcastReceivers, AlarmManager, or by abusing the device admin API. C2 channels include Firebase Realtime Database, Telegram bots, or encrypted HTTP. Tools like MobSF and Quark Engine help detect them.

♿ Accessibility Service – The Silent Hijacker

Click Automation & Screen Scraping

Accessibility Services can simulate user taps, read screen content, and even grant permissions without user consent (on older Android versions). This is the backbone of Android banking malware.

// Simulate a click
accessibilityEvent.performAction(AccessibilityNodeInfo.ACTION_CLICK);

Detection & Mitigation

Android 12+ restricts which apps can use Accessibility Services. Users must manually enable them. Apps can detect enabled accessibility services using `AccessibilityManager.getEnabledAccessibilityServiceList()`, but malware often uses generic names to evade this.

🏭 OEM & Vendor Attack Surfaces

Samsung Knox, Xiaomi, and Qualcomm Services

OEMs add custom services that often run with elevated privileges. Qualcomm's `com.qualcomm.qti.*` services have been exploited for kernel escalation. Samsung Knox offers a secure container, but bypasses exist via kernel exploits.

# List running services with their UID
adb shell dumpsys package | grep -A 5 "Service"

Pre‑installed Apps & Vendor Daemons

Carrier‑bloatware and system apps often have excessive permissions or debug interfaces. Use `adb shell pm list packages -f` to find hidden packages.

🔎 Android Forensics & Data Extraction

Logical & Physical Acquisition

Logical acquisition uses `adb backup` (if allowed) or content providers. Physical acquisition requires a forensic tool (e.g., Cellebrite) or a kernel exploit for a full memory dump.

adb backup -apk -shared -all -system -f backup.ab

SQLite Forensics & Keystore Artifacts

Extract `mmssms.db`, `contacts2.db`, and application databases. The Android Keystore stores keys in `/data/misc/keystore/`. Each key file is encrypted, but a kernel exploit can dump them from memory.

✍️ APK Signing & Trust Model

Signature Schemes v1‑v4 & Janus Vulnerability

Android supports multiple signature schemes. v1 (JAR signing) is vulnerable to the Janus attack (CVE‑2017‑13156), where a DEX file is added to an APK without invalidating the signature. Use apksigner to verify.

apksigner verify --verbose app.apk

Signature Spoofing & Downgrade Attacks

Some custom ROMs support signature spoofing for microG. Attackers can exploit this to install malicious updates. Always verify the certificate chain.

⛓️ Android Exploit Chains – The Full Workflow

Dirty Pipe → App Sandbox Escape → Kernel R/W → SELinux Bypass

Dirty Pipe (CVE‑2022‑0847) allowed writing to any file, including root‑owned files, from an untrusted app. This was chained with a kernel R/W primitive to disable SELinux, gaining full root access.

./exploit.sh  # CVE‑2022‑0847 PoC

Accessibility Abuse → Overlay Attack → Credential Theft → Persistent Malware

A malicious app requests Accessibility permissions. Once enabled, it overlays a fake login screen on top of the banking app, captures credentials, and installs a persistent backdoor via a BroadcastReceiver.

🛡️ RASP & App Hardening Techniques

Runtime Application Self‑Protection (RASP)

RASP libraries detect tampering, hooking, and emulation at runtime. They check for Frida artifacts, jailbreak files, and debugger attachment. Use DexGuard or Arxan for commercial RASP.

// Check for Frida port (conceptual)
if (socket.connect("127.0.0.1", 27042)) { abort(); }

Anti‑Frida & Anti‑Hooking Bypasses

Frida can be detected by scanning memory for its agent strings or by checking open ports. To bypass RASP, patch the detection functions using Frida itself or use a modified Frida server.

☁️ Mobile Backend & API Security

Firebase Exploitation & Token Lifecycle Abuse

Misconfigured Firestore rules allow unauthenticated read/write. Test with curl https://project.firebaseio.com/.json. JWT tokens can be intercepted and replayed if short‑lived or improperly validated.

GraphQL & REST API Mobile Attacks

Mobile apps often use the same APIs as the web version. Bypass SSL pinning with Frida, intercept traffic, and test for BOLA, excessive data exposure, and auth flaws just as you would on a web API.

📖 Real‑World Android Exploit Cases

StrandHogg (2019‑2020)

Task hijacking vulnerability that allowed malicious apps to overlay a fake login screen when the user tapped a legitimate app icon. It exploited Android's task affinity and `onTaskRemoved` handling. Affected all Android versions at the time. Fixed by introducing lockscreen verification for task switching.

Joker Malware (2017‑2022)

Stealthy malware that subscribed victims to premium SMS services. It bypassed Play Store protections by using DEX bytecode manipulation and executing in stages. It abused notifications to hide subscription confirmations. Google removed over 1,700 Joker‑infected apps from the Play Store.

Dirty Pipe (2022)

A kernel vulnerability (CVE‑2022‑0847) that allowed overwriting read‑only files, including root‑owned files. On Android, it was used to gain root access by overwriting `/etc/passwd` or injecting into system processes. Patched in Linux 5.16.11.

Lessons Learned

Layer your defenses. A single kernel bug or IPC flaw can defeat all higher‑level protections. Assume your app runs on a compromised device.

🧠 The Android Exploit Researcher's Mindset

“Where does Android place its trust?”

Trust in Android is layered: the kernel trusts the vendor, Binder trusts the caller, ART trusts the DEX bytecode, OEM services trust system privileges. Break any one of those, and you own the layer above.

“What if I spoof the caller's UID?”

Binder provides `getCallingUid()`. Some system services forget to validate it, allowing privilege escalation. Always check: does this endpoint trust the caller's identity implicitly?

The Golden Rule

Never test on a device you don't own. Never deploy exploits against live targets without explicit permission. Android security research is a serious responsibility.

🛡️ Advanced Android Defense & Hardening

SELinux Hardening & Hardware Attestation

Enforce SELinux in enforcing mode. Use hardware‑backed attestation for critical transactions. Never trust software‑only integrity checks. Implement app‑specific SELinux policies for system services.

Anti‑Hooking, RASP, and Secure IPC

Integrate RASP libraries that detect Frida, Xposed, and emulators. Use encrypted and signed Binder transactions where possible. Rotate secrets frequently.

🧪 Android Exploitation Labs

Binder Fuzzing Lab

Write a Python script using `frida` to call system services with malformed parcels and observe crashes in logcat. Focus on vendor‑specific services.

adb logcat -b crash | grep "binder"

Magisk Internals Lab

Patch a boot image manually with Magisk, understand the init hijacking process, and write a simple Zygisk module that hooks a system method.

SELinux Policy Analysis Lab

Extract the SELinux policy from a rooted device, decompile it with `sesearch`, and identify permissive domains that can be exploited.

adb pull /sys/fs/selinux/policy
sesearch -A policy | grep permissive

JNI Reversing & Exploitation Lab

Extract a native library from a target app, load it in Ghidra, find a vulnerable JNI function, and write a Frida hook to exploit it.

Lab Safety Rule

Use only dedicated test devices. Never connect a compromised device to a production network. Always wipe after experimentation.
